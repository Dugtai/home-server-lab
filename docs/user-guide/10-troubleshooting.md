# 10. Troubleshooting

Этот документ содержит базовую диагностику типовых проблем при настройке домашнего сервера-шлюза.

Цель: быстро определить, на каком уровне возникла ошибка — Proxmox, OpenWrt, DNS, Wi-Fi, VPN или anti-DPI.

## 1. Общий порядок диагностики

Всегда проверять от нижнего уровня к верхнему:

```text
1. Питание и кабели
2. Proxmox
3. OpenWrt VM
4. LAN / DHCP
5. DNS / AdGuard
6. WAN / Internet
7. VPN / PBR
8. anti-DPI
9. Клиентское устройство
```

Не нужно менять всё сразу. Нужно изменить один параметр, проверить результат и только потом двигаться дальше.

## 2. Нет доступа к Proxmox

Проверить:

- mini PC включён;
- Ethernet-кабель подключён в LAN-порт;
- клиент находится в той же сети;
- IP Proxmox не изменился;
- OpenWrt VM запущена, если Proxmox уже перенесён за OpenWrt;
- нет конфликта IP.

Локальная проверка через монитор и клавиатуру:

```bash
ip a
cat /etc/network/interfaces
systemctl status networking
```

Если сеть сломана после изменения `/etc/network/interfaces`, исправить файл локально и перезагрузить:

```bash
nano /etc/network/interfaces
reboot
```

## 3. OpenWrt VM не запускается

Проверить в Proxmox:

- VM создана;
- диск OpenWrt подключён;
- диск включён в Boot Order;
- RAM и CPU выделены;
- сетевые адаптеры подключены к правильным bridge;
- нет ошибок в консоли VM.

Ожидаемая схема:

```text
net0 -> vmbr0 -> WAN
net1 -> vmbr1 -> LAN
```

## 4. Клиенты не получают IP

Проверить:

- DHCP включён на OpenWrt LAN;
- клиент подключён к LAN-сегменту;
- access point работает в AP mode;
- DHCP на access point выключен;
- кабель LAN подключён в правильный порт;
- OpenWrt VM запущена.

На клиенте проверить IP.

Windows:

```cmd
ipconfig /all
```

Linux:

```bash
ip a
ip route
```

Ожидаемый результат:

```text
IP: 192.168.10.x
Gateway: 192.168.10.1
DNS: 192.168.10.3 after AdGuard setup
```

## 5. Интернет не работает, но IP есть

Проверить с клиента:

```bash
ping 192.168.10.1
ping 1.1.1.1
ping google.com
```

Интерпретация:

| Результат | Возможная причина |
|---|---|
| Не пингуется 192.168.10.1 | проблема LAN/OpenWrt |
| Пингуется 192.168.10.1, но не 1.1.1.1 | проблема WAN/NAT/firewall |
| Пингуется 1.1.1.1, но не google.com | проблема DNS |

## 6. DNS не работает

Проверить:

- AdGuard LXC запущен;
- IP AdGuard — `192.168.10.3`;
- порт 53 доступен;
- OpenWrt DHCP выдаёт DNS `192.168.10.3`;
- в AdGuard есть upstream DNS;
- запросы видны в Query Log.

Проверка с клиента:

```bash
nslookup google.com 192.168.10.3
```

Если AdGuard недоступен, временно можно вернуть DNS на OpenWrt или публичный DNS, чтобы восстановить интернет.

## 7. Wi-Fi работает нестабильно

Проверить:

- access point в режиме AP;
- DHCP на AP отключён;
- питание AP стабильно;
- если используется PoE, хватает ли мощности;
- уровень сигнала;
- не перегружен ли канал;
- не конфликтуют ли SSID 2.4 GHz и 5 GHz;
- клиенты получают IP от OpenWrt.

## 8. VPN не работает

Проверить:

- правильность времени на OpenWrt;
- VPN-интерфейс поднят;
- есть handshake;
- firewall zone настроена;
- PBR не направляет трафик ошибочно;
- endpoint доступен;
- ключи не перепутаны;
- AllowedIPs указаны корректно.

Важно: реальные VPN-ключи нельзя публиковать в GitHub.

## 9. Сайт должен идти через VPN, но идёт напрямую

Проверить:

- правило PBR включено;
- домен указан правильно;
- DNS-запрос идёт через ожидаемый DNS;
- домен не использует другой CDN;
- правило не перекрывается более приоритетным правилом;
- PBR-сервис перезапущен после изменений.

Иногда проще направить через VPN всё устройство, а не отдельный домен.

## 10. YouTube не работает

Проверить:

- DNS-запросы YouTube видны в AdGuard;
- домены YouTube попадают в anti-DPI обработку;
- видео не уходит случайно через VPN;
- работает ли `googlevideo.com`;
- нет ли проблемы с UDP 443 / QUIC;
- не перегружен ли CPU OpenWrt VM.

Проверять нужно не только главную страницу, но и запуск видео.

## 11. Discord не работает

Проверить:

- открывается ли приложение;
- работают ли текстовые каналы;
- грузятся ли картинки;
- работает ли voice;
- нет ли packet loss;
- UDP-трафик не блокируется firewall;
- Discord не ушёл случайно через медленный VPN;
- домены Discord попадают в нужную обработку.

## 12. После изменения настроек стало хуже

Правильный порядок восстановления:

1. Вернуть последнее изменение.
2. Перезапустить соответствующий сервис.
3. Проверить базовый интернет.
4. Проверить DNS.
5. Проверить VPN/PBR.
6. Только потом менять следующий параметр.

## 13. Минимальный набор команд диагностики

Proxmox:

```bash
ip a
ip route
cat /etc/network/interfaces
systemctl status networking
qm list
pct list
```

OpenWrt:

```bash
ip a
ip route
logread
/etc/init.d/network restart
/etc/init.d/firewall restart
```

Client:

```bash
ping 192.168.10.1
ping 1.1.1.1
nslookup google.com
traceroute google.com
```

## 14. Главное правило

Если что-то сломалось, не нужно сразу переустанавливать всё.

Сначала нужно понять, какой слой не работает:

```text
Physical -> Proxmox -> OpenWrt -> DHCP -> DNS -> WAN -> VPN/PBR -> anti-DPI -> client
```

## Итог

Этот документ должен использоваться как первая точка диагностики перед изменением конфигураций.
